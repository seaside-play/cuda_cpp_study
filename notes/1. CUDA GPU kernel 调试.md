工业界 CUDA cubin / GPU kernel 调试、排错主流方案（按使用频率排序）
工业场景特点：不依赖本地图形界面、兼顾性能与定位效率、兼顾线上线下、优先不关闭优化，极少用纯 -G 全调试模式。
一、线上 / 集群批量定位崩溃：cuda-memcheck + 错误埋点（最常用）
适用场景
推理 / 训练服务、大规模集群、GPU 崩溃、内存越界、空指针、shared memory 溢出、race condition。
标准流程
编译只加 -lineinfo（保留优化、保留行号，不损失性能）
bash
运行
nvcc -arch=sm_86 -lineinfo xxx.cu
线上复现程序套一层 cuda-memcheck 跑
bash
运行
cuda-memcheck --leak-check full --check-stack ./your_app
输出信息：出错线程、block/thread 索引、源码行、对应 SASS 地址，直接定位越界读写。
工业优势
无图形界面，服务器集群、docker、k8s 都能跑；
能批量跑离线复现脚本，自动化 CI 集成；
区分全局内存 / 共享内存 / 局部内存违规；
补充工具：compute-sanitizer（cuda-memcheck 升级版，CUDA 11.3+ 主推）
bash
运行
compute-sanitizer --tool memcheck ./app
支持竞争检测 racecheck、未初始化内存 initcheck，比旧 cuda-memcheck 精度更高，工业新环境统一用它。
二、性能瓶颈、寄存器 / 共享内存溢出静态分析：cuobjdump + nvdisasm（开发标配）
所有 GPU 开发团队必用离线工具，不需要运行 GPU，不用复现崩溃，提前规避性能 / 编译隐患。
cuobjdump 看 kernel 基础指标（CI 自动化校验）
bash
运行
# 查看每个kernel寄存器占用、smem大小、常量内存
cuobjdump -sass xxx.cubin
cuobjdump -ptx xxx.cubin
cuobjdump -symbols xxx.cubin
重点关注：
Registers/thread：寄存器过多会降低 occupancy，卡算力利用率；
Shared Memory 占用：超硬件上限直接运行时报错；
nvdisasm 反汇编 SASS 机器码，定位低效指令
bash
运行
nvdisasm -g xxx.cubin > sass_dump.txt
工业用途：
排查重复全局内存读写、缺少 ldg/stg 缓存、分支发散；
核对向量化加载、Tensor Core 指令是否真正生成；
对比优化前后汇编，验证算子优化效果；
无源码只有 cubin 时逆向分析第三方算子。
三、线下开发、算法算子单步调试：Nsight Compute（算法工程师主力）
使用场景
本地开发、算子逻辑错误、数值异常、warp 分支、访存延迟分析。
工业标准编译参数（两套分开）：
调试版本（临时，只本地用）：-G -g 关优化，断点看变量
性能分析版本（对标线上）：-lineinfo 保留优化，做 profiling
核心工业用法：
单步 GPU 线程，切换 block/warp/thread；
同时显示 C++ 源码 + PTX + SASS 三对照；
采样 SM 利用率、LD/ST 吞吐、shared bank conflict；
导出报告给性能优化团队归档；
局限
生产服务器无桌面无法使用，仅本地开发机、图形工作站使用。
四、线上崩溃日志定位：CUDA Runtime/Driver 错误码 + 同步埋点（线上必备）
线上环境不能开 sanitizer（性能损耗极高），靠规范埋点捕获 cubin 加载、kernel 执行报错。
加载 cubin 阶段校验 Driver API 返回码
cuModuleLoad / cuModuleGetFunction / cuLaunchKernel 全部捕获 CUresult，打印错误字符串；
常见工业报错：
CUDA_ERROR_INVALID_IMAGE：cubin sm 与硬件不匹配；
CUDA_ERROR_OUT_OF_MEMORY：常量 / 共享内存超限；
Kernel 执行后强制同步并捕获异步错误
cpp
运行
cudaDeviceSynchronize();
auto err = cudaGetLastError();
if (err != cudaSuccess) {
    LOG_ERROR << "kernel launch fail:" << cudaGetErrorString(err);
}
崩溃时开启 CUDA 核心转储
bash
运行
export CUDA_ENABLE_COREDUMP_ON_EXCEPTION=1
捕获 GPU 异常现场，事后离线解析。
五、多卡 / 分布式、自动化回归：NSight CLI（无图形界面批量 Profiling）
Nsight Compute 提供命令行模式，集成 CI / 自动化测试流水线，工业大规模算子验证标配：
bash
运行
ncu --export profile_report ./your_app
可批量采集：occupancy、访存延迟、指令吞吐量、Tensor Core 利用率，自动生成 JSON 报告，用于自动化性能卡点。
六、极少数场景：cuda-gdb 纯命令行断点调试
仅部分底层驱动、自定义 runtime 团队使用，算法团队很少用：
无 GUI 工作站需要源码断点；
手动 Driver API 加载 cubin 自定义运行时；
缺点：多线程多 block 切换繁琐，远不如 Nsight 直观。
工业界分层使用总结（团队标准流程）
1. 编码阶段（开发本地）
静态检查：cuobjdump / nvdisasm 提前看寄存器、smem、SASS；
逻辑排错：Nsight Compute（-G 模式断点调数值）；
性能调优：Nsight Compute（-lineinfo 性能剖面）。
2. CI 自动化流水线（每次提交必跑）
compute-sanitizer 跑内存越界、数据竞争检测；
ncu 采集性能指标，低于阈值阻断合并；
cuobjdump 自动校验寄存器占用，防止 occupancy 暴跌。
3. 线上生产环境（服务器 / 容器）
全链路错误埋点，捕获 cubin 加载、kernel 异步错误；
崩溃开启 CUDA coredump，事后离线复现；
出现偶现崩溃时，离线环境用 compute-sanitizer 复现定位。
淘汰 / 极少使用的方式
纯 cuda-gdb 日常调试：效率太低；
线上长期开 -G：关闭优化，性能暴跌，禁止上线；
只靠 printf 大量打印：吞吐极低，仅临时简单调试，不能用于排查复杂越界 / 竞争。
